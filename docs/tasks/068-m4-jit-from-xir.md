# M4：JIT 从 Xi IR 接入 — 详细实施计划

## 1. 现状分析

### 1.1 当前 JIT 编译流水线

```text
AST → xi_lower → XiFunc (SSA IR) → xi_emit → XrProto (bytecode)
                                                  ↓ (XiFunc 立即释放)
                                            VM 解释执行
                                                  ↓ (热函数检测: call_count > threshold)
                                            xir_build_from_proto_jit()
                                                  ↓ (bytecode → XIR SSA 重建)
                                            xir_run_pipeline_ex() (优化)
                                                  ↓
                                            xir_codegen_arm64/x64() (机器码)
```

**核心浪费**：Xi IR 已经是 SSA 形式，但 `xi_emit` 将它扁平化为 bytecode 后立即丢弃。
JIT 的 `xir_builder` (7980 行) 从 bytecode 重新推断 SSA——这是重复劳动。

### 1.2 Builder 职责分解

`xir_builder*.c` 共 7661 行 C 代码（不含头文件），做了以下工作：

| 职责 | 行数估算 | Xi IR 是否已提供 |
|------|---------|----------------|
| CFG 构建 (bb_leaders → XirBlock) | ~200 | ✅ Xi IR 有 XiBlock + CFG |
| Braun SSA 构建 (slot→vreg、phi) | ~500 | ✅ Xi IR 已是 SSA |
| 202 个 OP_* 翻译 case | ~5000 | ⚠️ 需要 Xi→XIR op 映射 |
| 类型追踪 (slot_tag, slot_rep) | ~400 | ✅ Xi IR 每个 XiValue 有 XrType* |
| IC 反馈集成 (field/method/builtin) | ~800 | ❌ 运行时数据，不在编译时 |
| Deopt 快照生成 | ~300 | ❌ 需要 SSA→bytecode slot 映射 |
| 异常处理 (try/catch 栈) | ~200 | ⚠️ Xi IR 有 XI_TRY/XI_CATCH |
| OSR 支持 (loop header 标记) | ~100 | ⚠️ Xi IR 有 is_loop_header |

**结论**：~1200 行 (CFG + SSA + 类型) 可以直接消除，~5000 行 op 翻译需要新的映射函数，
~1100 行 (IC + deopt) 必须保留但可简化。

### 1.3 关键数据结构对照

| JIT Builder 概念 | 当前来源 (XrProto) | Xi IR 等价物 |
|-----------------|-------------------|-------------|
| `bb_leaders` bitmap | 编译器生成 | `XiFunc.blocks[]` |
| `slot_map[256]` (reg→vreg) | 运行时构建 | `XiValue` 即 SSA def |
| `param_types[i]` | `proto->param_types` | `XiFunc.params[i]->type` |
| `inst_types[pc]` | 仅 legacy 编译器 | `XiValue.type` (每个值自带) |
| `type_feedback` | 运行时 IC | 运行时 IC (不变) |
| `Blueprint.loops` | 编译器生成 | `XiBlock.is_loop_header` |
| `shared_protos[i]` | 运行时构建 | `XiFunc.children[]` |
| `deopt_info.bc_pc` | bytecode PC | ❌ 需要映射表 |

### 1.4 核心挑战：Deopt

JIT deopt 将机器码状态还原为 VM bytecode 帧。这需要：
1. **bytecode PC**：deopt 目标在哪条 bytecode 指令恢复
2. **slot mapping**：每个 XIR vreg 对应哪个 bytecode slot (R[0..255])
3. **类型标注**：每个 slot 的 XR_TAG (I64/F64/PTR/TAGGED)

目前 `xi_emit.c` 内部的 `reg_map[value_id] → register` 映射在 emit 完成后丢弃。
**这是 M4 的第一个必须解决的问题**。

## 2. 总体策略：渐进迁移

```text
阶段 0: 保留 Xi IR + 生成 deopt 映射表 (XiSlotMap)       ← 无行为变化
阶段 1: 新建 xi_to_xir.c，先覆盖算术/比较/分支/phi    ← 双路径对比
阶段 2: 覆盖函数调用/返回/闭包/upvalue               ← 双路径对比
阶段 3: 覆盖属性访问/方法调用/IC 集成                 ← 双路径对比
阶段 4: 覆盖 array/map/exception/coroutine            ← 双路径对比
阶段 5: 删除 xir_builder*.c (7980 行)                 ← 清理
```

每个阶段都是独立可交付的 commit，全量测试通过后才进入下一阶段。

## 3. 阶段 0：保留 Xi IR + Deopt 映射表

### 3.1 目标

- 在 `xi_emit` 过程中导出 `XiSlotMap`（value_id → bytecode slot + PC 映射）
- 将 XiFunc* 存活期延长到 XrProto 生命周期（挂在 proto 上）
- 无行为变化，所有测试不受影响

### 3.2 XiSlotMap 结构

```c
/* Mapping from Xi IR value ID to bytecode slot, generated by xi_emit.
 * Enables JIT to generate deopt snapshots from Xi IR values instead
 * of bytecode re-analysis. */
typedef struct {
    uint16_t value_id;   /* Xi IR value ID (XiValue.id) */
    uint8_t  bc_slot;    /* bytecode register R[0..255] */
    uint8_t  xr_tag;     /* XR_TAG_* at this point */
    uint32_t bc_pc;      /* bytecode PC where this value is live */
} XiSlotMapEntry;

typedef struct XiSlotMap {
    XiSlotMapEntry *entries;
    uint32_t count;
    uint32_t capacity;
} XiSlotMap;
```

### 3.3 修改 xi_emit.c

在 `emit_value()` 中，每当 `reg_map[v->id] = reg` 被写入时，同时记录到 `XiSlotMap`。
emit 完成后将 `XiSlotMap` 存入 `XrProto`。

### 3.4 延长 XiFunc 生命周期

当前 `xcompiler.c` 编译完成后立即调用 `xi_pipeline_result_free()`，释放 `XiFunc`。

修改策略：
- 在 `XrProto` 新增 `void *xi_func_ref` (opaque，避免暴露 Xi IR 头文件到 runtime)
- `xi_pipeline_result_free()` 不再释放 `res->ir`；改由 `proto` 接管所有权
- `xr_proto_free()` 中释放 `xi_func_ref`

**内存开销估算**：每个 XiFunc 约 2-10KB（取决于函数大小），典型应用 100 个函数 ≈ 1MB。
可接受——JIT 热函数通常只占 10-20%，且编译后可选择释放。

### 3.5 交付物

- `xi_emit.c`：生成 XiSlotMap，存入 proto
- `xchunk.h`：新增 `xi_func_ref`、`xi_slot_map` 字段
- `xchunk.c`：释放逻辑
- `xcompiler.c`：不再立即释放 XiFunc
- 验证：96/96 ctest，293+ regression，23+ AOT

## 4. 阶段 1：Xi IR → XIR 翻译器 (算术/比较/分支/phi)

### 4.1 新文件：`src/jit/xi_to_xir.c` + `src/jit/xi_to_xir.h`

```c
/* Xi IR → XIR lowering for JIT compilation.
 * Translates Xi SSA (XiFunc) to JIT SSA (XirFunc) without
 * going through bytecode reconstruction.
 *
 * Advantages over bytecode builder:
 * - No SSA re-construction (Xi IR already has SSA + PHI)
 * - Precise types on every value (XrType* vs slot_tag heuristic)
 * - Direct access to Xi IR metadata (class data, closures)
 */

XR_FUNC XirFunc *xi_to_xir_lower(XiFunc *xi_func, XrProto *proto,
                                   XiSlotMap *slot_map,
                                   struct XrayIsolate *isolate);
```

### 4.2 翻译映射 (阶段 1 覆盖)

| Xi IR Op | XIR Op | 备注 |
|----------|--------|------|
| `XI_CONST_INT` | `xir_const_i64()` | 直接映射 |
| `XI_CONST_FLOAT` | `xir_const_f64()` | 直接映射 |
| `XI_CONST_BOOL` | `xir_const_i64(0/1)` | rep=I64 |
| `XI_CONST_NIL` | `xir_const_i64(0)` + tag=NULL | |
| `XI_ADD/SUB/MUL/DIV/MOD` | `XIR_ADD/SUB/MUL/DIV/MOD` | 按 type 选 rep |
| `XI_NEG/XI_NOT` | `XIR_NEG/XIR_NOT` | |
| `XI_EQ/NE/LT/LE/GT/GE` | `XIR_CMP_*` | |
| `XI_BAND/BOR/BXOR/BNOT` | `XIR_AND/OR/XOR/NOT` | |
| `XI_SHL/SHR` | `XIR_SHL/XIR_SHR` | |
| `XI_BR` | `XIR_JMP` | |
| `XI_BR_COND` | `XIR_JCC` | |
| `XI_RETURN/RETURN_VOID` | `XIR_RET` | |
| `XI_PHI` | `XirPhi` | 直接映射 args |

### 4.3 CFG 映射

```text
for each XiBlock in xi_func:
    create XirBlock
    map XiBlock.id → XirBlock.id
    copy predecessor/successor edges
    for each XiPhi in XiBlock:
        create XirPhi with mapped args
    for each XiValue in XiBlock:
        translate to XIR instruction
```

### 4.4 类型 → Rep 映射

```c
static uint8_t xi_type_to_xir_rep(XrType *type) {
    if (!type) return XR_REP_I64;  /* tagged fallback */
    switch (type->kind) {
        case XR_KIND_INT:    return XR_REP_I64;
        case XR_KIND_FLOAT:  return XR_REP_F64;
        case XR_KIND_BOOL:   return XR_REP_I64;
        default:             return XR_REP_I64;  /* tagged */
    }
}
```

### 4.5 Deopt 生成

使用阶段 0 的 XiSlotMap：
```c
/* For each guard point (type check, overflow, etc.):
 * 1. Look up current Xi IR value in slot_map → bc_slot
 * 2. Create XirDeoptInfo { bc_pc, slots[] }
 * 3. Attach to guard instruction */
```

### 4.6 双路径验证

在 `xir_jit_try_compile` 中，同时运行新旧路径，比较 XirFunc 输出：

```c
#ifdef XI_JIT_DUAL_PATH
    XirFunc *old_func = xir_build_from_proto_jit(...);
    XirFunc *new_func = xi_to_xir_lower(xi_func, proto, slot_map, isolate);
    /* Compare block count, vreg count, instruction sequence */
    xi_to_xir_validate(old_func, new_func);
    xir_func_destroy(new_func);  /* use old path for execution */
#endif
```

### 4.7 交付物

- `src/jit/xi_to_xir.c` (~600 行)：阶段 1 翻译
- `src/jit/xi_to_xir.h`：公共 API
- 验证：全量测试通过 + 双路径比较 0 差异

## 5. 阶段 2：函数调用 / 返回 / 闭包

### 5.1 翻译映射

| Xi IR Op | XIR Op | 备注 |
|----------|--------|------|
| `XI_CALL` | `XIR_CALL_C(xr_jit_call)` | JIT call bridge |
| `XI_CALL_KNOWN` | `XIR_CALL_KNOWN` | 已知 proto，可内联 |
| `XI_CLOSURE_NEW` | `XIR_CALL_C(xr_jit_closure_new)` | |
| `XI_RETURN` | `XIR_RET` + store return value | |
| `XI_GETSHARED/XI_SETSHARED` | load/store from shared array | |
| `XI_GET_UPVAL/XI_SET_UPVAL` | load/store upvalue cell | |

### 5.2 Call Argument Pool

Xi IR 的 `XI_CALL` 参数在 `v->args[]` 中。直接映射到 XIR 的 `call_arg_pool`：

```c
XirRef *xir_args = ...;
for (int i = 0; i < xi_call->nargs; i++)
    xir_args[i] = xi_to_xir_ref(xi_call->args[i]);
builder_bind_call_args(func, call_dst, xir_args, nargs);
```

### 5.3 IC 反馈集成

对 `XI_CALL` / `XI_CALL_METHOD`：
- 读 `proto->type_feedback` 中对应 bytecode PC 的 IC 数据
- 通过 `XiSlotMap` 查找 Xi IR value 对应的 bytecode PC
- 使用 IC 数据决定是否 emit GUARD_TYPE + CALL_KNOWN

- `xi_to_xir.c` 扩展到 ~1200 行
- 验证：双路径比较 + 全量测试

## 7. 阶段 4：Array/Map/Exception/Coroutine

### 7.1 翻译映射

| Xi IR Op | XIR Op |
|----------|--------|
| `XI_ARRAY_NEW/PUSH/GET/SET` | `XIR_CALL_C` bridges |
| `XI_MAP_NEW/GET/SET` | `XIR_CALL_C` bridges |
| `XI_TRY/XI_CATCH/XI_THROW` | `XIR_CALL_C` + deopt |
| `XI_YIELD/XI_AWAIT` | `XIR_SUSPEND` + resume entry |
| `XI_CLASS_CREATE` | `XIR_CALL_C` |
| `XI_INSTANCE_NEW` | `XIR_CALL_C` |

### 7.2 Coroutine Resume

Xi IR 的 `XI_YIELD` / `XI_AWAIT` 对应 `XIR_SUSPEND`。
每个 suspend point 需要 resume entry（机器码地址），由 codegen 生成。
Xi IR 中 suspend 后的 continuation block 直接映射为 resume 目标。

## 9. 风险与缓解

| 风险 | 影响 | 缓解 |
|------|------|------|
| XiFunc 内存增长 | ~1MB for 100 functions | 仅保留热函数的 XiFunc；JIT 编译后释放 |
| Deopt 精度不足 | JIT→VM 恢复失败 | XiSlotMap 在 xi_emit 中精确记录；单元测试覆盖 |
| IC 反馈 PC 映射 | Xi IR 没有 bytecode PC | 通过 XiSlotMap 的 bc_pc 字段间接映射 |
| 双路径性能开销 | 编译时间翻倍 | 仅 debug build 启用；release build 只走新路径 |
| Inlining 不兼容 | callee 需要 XiFunc | 确保 callee proto 也保留 xi_func_ref |
