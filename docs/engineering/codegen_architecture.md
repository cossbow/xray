# xray Codegen 架构

> **作者**: Cascade (pair-programming with Xinglei Xu)
> **日期**: 2026-05-12
> **状态**: accepted
> **配套实施 task**: [`../tasks/073-codegen-hardening.md`](../tasks/073-codegen-hardening.md)
> **路线选型**: 曾考虑过 codegen DSL 方案（参考 LLVM TableGen / Cranelift ISLE），评估后认为不适合 xray 当前规模与目标，未被采纳。详见 §1.4 / §1.5

## 摘要

xray JIT codegen 选择**手写 assembler class + 自写 disassembler + 强测试**的工程化路线（Dart / V8 / HotSpot / SpiderMonkey 同款），不引入 codegen DSL。本文档定义这条路线的架构契约：driver / assembler 双层分工、disassembler 是 first-class 工程资产、测试用 disassembly golden 而非 byte hex。

---

## 1. 现状与选型

### 1.1 当前 codegen 资产

xray 实际上已经实现了 Dart 风格架构的多数底层组件：

| 资产 | 文件 | 规模 | 性质 |
|---|---|---|---|
| x64 byte-stream assembler | `src/jit/xm_x64.{h,c}` | ~20K | `X64Buf` + emit helpers (`x64_emit8` / `x64_modrm` / `x64_rex` / ...) |
| arm64 fixed-width assembler | `src/jit/xm_arm64.{h,c}` | ~22K | encode 单条 ARM64 指令为 `uint32_t` |
| arm64 disassembler | `src/jit/xm_arm64_disasm.h` | ~18K | header-only pure func |
| x64 codegen driver | `src/jit/xm_codegen_x64*.c` | ~5.1K | `Xm` IR → x64，调用 assembler |
| arm64 codegen driver | `src/jit/xm_codegen*.c` (含 mem/call/stub) | ~5.8K | `Xm` IR → arm64，调用 assembler |
| JIT E2E test | `tests/unit/jit/test_jit_e2e.c` | 单文件 | 行为级测试 |

driver / assembler 双层分工已经是事实，现状架构本身是健康的。

### 1.2 缺口

补齐工业级工程化所欠缺的能力：

| 缺口 | 影响 |
|---|---|
| x64 disassembler | crash / dump / debug 时无法在 xray 内部反汇编 x64 字节 |
| byte-level emit unit test framework | 没有 Dart 的 `ASSEMBLER_TEST_GENERATE/RUN` 等价物，emit 正确性靠 E2E test 间接验证 |
| codegen fail-fast 契约 | `xm_codegen_x64*.c` 当前**无任何** `XR_UNREACHABLE` / `XR_FATAL` 拦截未处理路径 — 这就是 commit `8b06997` "RT-opcode 漏 emit" 的根因 |
| `llvm-mc` differential CI | 没有对 generated bytes 与业界标准 reference 的 cross-validation |
| codegen fuzz | 没有 random program generator 持续打 codegen edge case |
| arm64 disassembler 与 assembler 的 round-trip test | `xm_arm64_disasm.h` 存在但未被测试覆盖固化 |

### 1.3 直接动机

commit `8b06997` 修复的 bug：x64 codegen 在 RT-opcode fallback 路径**漏 emit 任何字节**而 arm64 同位置 emit 了一个守护 NOP，导致 Win64 上出现 `STATUS_HEAP_CORRUPTION`。

根因分析：
- codegen driver 中的某些 switch / if-else 没有强制 `default: XR_UNREACHABLE`
- 没有 byte-level unit test 直接验证"每个 IR op → emit ≥ 1 字节"
- 没有 fuzz / differential 持续打这类 edge case

这三条都不需要 DSL 才能解决。

### 1.4 工业界对比

| 项目 | Codegen 路线 | Backend 数 | 规模 | 有 DSL? |
|---|---|---|---|---|
| **Dart VM** | 手写 assembler class + 自写 disassembler | 5 (arm/arm64/ia32/x64/riscv) | ~74K | ❌ |
| **V8** | 手写 MacroAssembler | 5+ | ~150K+ | ❌ |
| **HotSpot (OpenJDK)** | 手写 MacroAssembler | 6+ | ~200K+ | ❌ |
| **SpiderMonkey** | 手写 MacroAssembler | 4 | ~80K+ | ❌ |
| **LuaJIT** | DynASM (runtime DSL) | 5 | 极小 | ✅ |
| **LLVM** | TableGen (full backend framework) | 全栈 | 巨大 | ✅ |
| **Cranelift** | ISLE (rewrite system) | 4 | 中 | ✅ |

> 工业界主流 JIT（Dart / V8 / HotSpot / SpiderMonkey）**全部手写**。**写 DSL 的是 backend framework 厂（LLVM / Cranelift），不是 JIT 厂**。xray 是 JIT，不是 backend framework。

### 1.5 选型结论

**xray 选择 Dart 风格工程化**：补齐 §1.2 缺口，不引入 codegen DSL。

理由：
- xray 当前 codegen ~12K 行，远未达到 V8 / HotSpot 量级；现状 assembler 抽象已经足够
- xray 团队规模小，DSL 引入额外的 generator + schema 维护是负担
- Dart 实证 5 个 backend、74K 行手写依然可维护
- 真正的 robustness gap 在 fail-fast 契约 + 测试覆盖，DSL 也解决不了 runtime-sensitive 路径
- 不引入 DSL 节省的精力可以投入到 disassembler / fuzz / parity test 这类高 ROI 工程

## 2. 核心决策

### 2.1 不引入 codegen DSL

**决策**：不写 `.isa` schema，不写 generator，不引入 build-time 代码生成。

**理由**：
- xray 规模在 Dart 实证可控范围内
- DSL v1 不能表达 runtime-sensitive paths（call / deopt / OSR / safepoint），这些仍要手写，DSL 收益打折
- DSL 引入永久维护成本（generator 自身 ~2K 行 C），ROI 低于直接补 assembler 测试

### 2.2 不引入 instruction scheduler / pattern matcher

**决策**：codegen 保持 1:1 IR → machine code 风格，不做 multi-instruction fusion / latency-aware scheduling。

**理由**：
- xray JIT 优化深度浅，scheduler 收益有限
- x64 OoO 强、arm64 移动端有限场景才需要 scheduler，目前不在路径上
- 未来真有需求时是独立问题，不影响 codegen 架构

### 2.3 driver / assembler 双层架构

**决策**：保持现状的 driver / assembler 分层，spec 化职责边界。

| 层 | 文件 | 职责 |
|---|---|---|
| **assembler** | `xm_x64.{h,c}` / `xm_arm64.{h,c}` / 未来 `xm_riscv64.{h,c}` | 单条机器指令 encoding；纯函数；无 IR 概念 |
| **driver** | `xm_codegen_x64*.c` / `xm_codegen.c` 系列 | `Xm` IR → 选择 assembler 调用 + 寄存器分配出口 + patch 协议 + runtime 协议 |
| **runtime helpers** | `xm_jit_runtime*.c` | call ABI / deopt / OSR / safepoint / suspend / coroutine resume |

assembler 层内零 driver knowledge，driver 层不直接拼字节。

### 2.4 disassembler 是 first-class 工程资产

**决策**：`xm_<arch>_disasm.h` 是 codegen 架构的必需品而非可选 debug 工具。

**理由**：
- 调试 / dump / crash analysis 都需要它
- 测试框架 `EXPECT_DISASSEMBLY("add x0, x1, x2")` 依赖它
- arm64 已有先例，x64 / riscv64 必须补齐到同等水平

**实现选择**：自写而非嵌入第三方库（capstone）。理由：
- arm64 已经自写，跨 backend 一致性比代码量重要
- xray 已 ship `xm_arm64_disasm.h` 这种风格，新人 onboarding 路径已经存在
- capstone 二进制 ~500K，xray 的 release size constraint 不希望引入
- 自写 x64 / riscv64 disassembler 各 1-2K 行，可控

### 2.5 测试用 disassembly golden 而非 byte hex

**决策**：assembler unit test 的 expected 写成可读的 asm string，不写 hex bytes。

```c
ASSEMBLER_TEST_GENERATE(AddRegReg, asm) {
    a64_add_rr(asm, A64_X0, A64_X1, A64_X2);
    a64_ret(asm);
}
ASSEMBLER_TEST_RUN(AddRegReg, test) {
    EXPECT_DISASSEMBLY(
        "add x0, x1, x2\n"
        "ret\n");
}
```

vs 反例：

```c
EXPECT_BYTES(0x20, 0x00, 0x02, 0x8b,  /* unreadable */
             0xc0, 0x03, 0x5f, 0xd6);
```

**理由**（学 Dart）：
- 可读性高，code review 时一眼看出意图
- 维护性好，`add x0, x1, x2` 改成 `add x0, x1, x3` 不需要重算 hex
- 同时验证 encoding **和** disassembler 正确（双向 round-trip）
- 对照 LLVM `llvm-mc -disassemble` 输出做交叉验证更直接

## 3. 整体架构

### 3.1 三层结构

```text
┌──────────────────────────────────────────────────────────────┐
│  driver  (xm_codegen.c, xm_codegen_x64*.c, xm_codegen_call.c) │
│    Xm IR → 寄存器分配 → patch / OSR / deopt / safepoint        │
└─────────────────────┬────────────────────────────────────────┘
                      │ 调用 assembler API
                      ▼
┌──────────────────────────────────────────────────────────────┐
│  assembler  (xm_x64.{h,c}, xm_arm64.{h,c})                    │
│    单条机器指令 encoding；纯函数；无 IR / runtime knowledge    │
└─────────────────────┬────────────────────────────────────────┘
                      │ 写入 X64Buf / A64Buf
                      ▼
┌──────────────────────────────────────────────────────────────┐
│  disassembler  (xm_x64_disasm.h, xm_arm64_disasm.h)           │
│    bytes → text；header-only pure func；test / debug 复用      │
└──────────────────────────────────────────────────────────────┘
```

runtime helpers (`xm_jit_runtime*.c`) 与 assembler 同级独立。

### 3.2 文件布局

```text
src/jit/
├── xm.{h,c}                    # IR core

├── xm_codegen.{c,h}            # arm64 driver entry
├── xm_codegen_x64*.c           # x64 driver
├── xm_codegen_*.c (mem/call/stub/ins) # arm64 driver 子模块
├── xm_jit_runtime*.c           # runtime helpers
└── ...

tests/unit/jit/
```

| coverage script | `scripts/codegen_coverage_<arch>.sh` | per-arch |

## 4. Assembler 抽象

### 4.1 设计原则

1. **纯函数 / 无全局状态**：所有 emit fn 接受 `<arch>Buf*` 输出参数
2. **buffer overflow 检查内置**：`x64_emit8` / `x64_emit32` 内部 `XR_DCHECK(pos < cap)`
3. **每条 emit fn 至少写 ≥ 1 字节**：assembler 层无 "no-op emit" 概念
4. **共享 encoding 原语**：ModR/M / SIB / REX (x64) 与 bitfield (arm64) 走 helper，不在调用处手拼
5. **特殊寄存器 corner case 内置**：`rsp`/`rbp`/`r12`/`r13` 的 SIB / disp 规则在 `x64_modrm_mem` 里处理，调用方不感知
6. **签名稳定**：assembler API 不持有 IR / register allocator state，签名只接受 raw register / immediate

### 4.2 X64Buf 与 A64Buf 现状形状

xray 已有的 API（保持不变，spec 化为 contract）：

```c
/* xm_x64.h */
typedef struct {
    uint8_t  *code;
    uint32_t  pos;
    uint32_t  capacity;
} X64Buf;

void     x64_buf_init(X64Buf *buf, uint8_t *mem, uint32_t cap_bytes);
void     x64_emit8(X64Buf *buf, uint8_t b);
void     x64_emit32(X64Buf *buf, uint32_t val);
void     x64_emit64(X64Buf *buf, uint64_t val);
uint32_t x64_buf_offset(X64Buf *buf);
void     x64_rex(X64Buf *buf, bool w, bool r, bool x, bool b);
void     x64_modrm(X64Buf *buf, uint8_t mod, uint8_t reg, uint8_t rm);
void     x64_modrm_mem(X64Buf *buf, X64Reg reg, X64Reg base, int32_t disp);
void     x64_sib(X64Buf *buf, uint8_t scale, uint8_t index, uint8_t base);
/* ... 各类指令 emit fn */
```

```c
/* xm_arm64.h */
typedef struct {
    uint32_t *code;     /* fixed-width 4 bytes per insn */
    uint32_t  count;
    uint32_t  capacity;
} A64Buf;
/* ... encode fn 返回 uint32_t 或写 buf */
```

### 4.3 强 emit 契约

每条 assembler 顶层 emit fn 满足：

| 契约 | 实现 |
|---|---|
| 写入字节数 ≥ 1 | 由实现保证，无 "0-byte emit" 路径 |
| buffer 不溢出 | `x64_emit*` 内部 `XR_DCHECK(pos < capacity)` |
| 输入 reg / imm 合法 | `XR_DCHECK(reg < 16, ...)` / `XR_DCHECK(imm fits range)` |
| 不依赖 IR / runtime state | API 签名只接受 raw value |

driver 层对 assembler 的调用，必须满足：

| 契约 | 实现 |
|---|---|
| switch on `XmOp` 必须穷尽 | 末尾 `default: XR_UNREACHABLE("unhandled XmOp %d", op);` |
| codegen 入口的字节数自检 | driver 在 emit 完成后 `XR_DCHECK(buf.pos > entry_pos);` 或对每个 IR op 包一层 `emit_with_check()` 验证 ≥ 1 字节 |
| 任何 fallback 路径必须**显式** emit 或 `XR_UNREACHABLE` | 静默 fallback 是 commit `8b06997` 的根因，禁止 |

### 4.4 共享原语

x64 共享：
- `x64_rex` / `x64_rex_rr`：REX prefix 计算
- `x64_modrm` / `x64_modrm_rr` / `x64_modrm_mem`：ModR/M 计算（含 rsp/rbp/r12/r13 special case）
- `x64_sib`：SIB 字节
- `x64_patch_rel32`：rel32 patch helper

arm64 共享：
- bitfield 编码 helper（位拼接 / 符号扩展 / log2 imm）
- condition code 表
- branch reach 计算

新增指令时优先复用 helper，禁止在 emit fn 里重复手拼。

## 5. Disassembler

### 5.1 接口形状

```c
/* xm_<arch>_disasm.h */

/* Disassemble one instruction starting at code[offset].
 * Writes human-readable asm into buf (must be ≥ 64 bytes).
 * Returns the number of bytes consumed (1-15 for x64, always 4 for arm64).
 * Returns 0 if instruction not recognized; caller should display "<unknown>".
 */
size_t x64_disasm_one(const uint8_t *code, size_t code_size,
                       size_t offset, char *buf, size_t buf_size);

size_t a64_disasm_one(uint32_t insn, char *buf, size_t buf_size);

/* Disassemble a code region into multiline text. */
void x64_disasm_region(const uint8_t *code, size_t size,
                        char *buf, size_t buf_size);
```

### 5.2 实现规范

| 规范 | 理由 |
|---|---|
| header-only pure func | 与 `xm_arm64_disasm.h` 一致；test 复用零成本 |
| 输出风格学 GNU AT&T 或 ARM 官方 | 让 `EXPECT_DISASSEMBLY` 可复用 `llvm-mc` 输出做对比 |
| 不要求覆盖整个 ISA | 只覆盖 xray JIT assembler 实际 emit 的指令子集 |
| 未识别指令返回 0 / 写 `"<unknown>"` | crash dump 时退化优雅 |
| 与 assembler 同 backend round-trip 测试 | 每条 assembler emit 都有反汇编断言 |

### 5.3 与 LLVM `llvm-mc` 的关系

| 用途 | 用谁 |
|---|---|
| 运行时 dump / crash report | xray 自己的 `xm_<arch>_disasm.h` |
| CI cross-validation | `llvm-mc -disassemble` (作为第三方 reference) |
| 测试 expected string 的 ground truth | `llvm-mc` 离线生成，commit 进 repo |

**xray 二进制只链接自家 disassembler，不运行时依赖 llvm-mc**。

## 6. 测试架构

### 6.1 ASSEMBLER_TEST 框架

学 Dart `runtime/vm/compiler/assembler/assembler_test.h`：

```c
/* tests/unit/jit/assembler_test.h */

#define ASSEMBLER_TEST_GENERATE(name, asm)                       \
    static void asm_test_generate_##name(<arch>Buf *asm)

#define ASSEMBLER_TEST_RUN(name, test)                           \
    static void asm_test_run_##name(AssemblerTestCtx *test);     \
    static void asm_test_##name(void) {                          \
        AssemblerTestCtx ctx;                                    \
        assembler_test_ctx_init(&ctx);                           \
        asm_test_generate_##name(&ctx.asm);                      \
        asm_test_run_##name(&ctx);                               \
        assembler_test_ctx_destroy(&ctx);                        \
    }                                                            \
    static void asm_test_run_##name(AssemblerTestCtx *test)

#define EXPECT_DISASSEMBLY(expected_str)                         \
    do {                                                         \
        char actual[1024];                                       \
        assembler_test_disassemble(test, actual, sizeof(actual)); \
        if (strcmp(actual, expected_str) != 0) {                 \
            xr_test_fail("disassembly mismatch:\n"               \
                         "  expected:\n%s\n"                     \
                         "  actual:\n%s\n",                      \
                         expected_str, actual);                  \
        }                                                        \
    } while (0)
```

`AssemblerTestCtx` 内部持有 `<arch>Buf` + 测试用 code memory + disassembler scratch buffer。

测试用例形式：

```c
ASSEMBLER_TEST_GENERATE(MovzImm16, asm) {
    a64_movz(asm, A64_X0, 42, 0);
    a64_ret(asm);
}
ASSEMBLER_TEST_RUN(MovzImm16, test) {
    EXPECT_DISASSEMBLY(
        "movz x0, #0x2a\n"
        "ret\n");
    /* 行为验证（可选）：在真机上跑，看返回值 */
    EXPECT_INT64_RETURN(test, 42);
}
```

### 6.2 Coverage 目标

| Backend | 文件 | 目标 |
|---|---|---|
| x64 | `tests/unit/jit/test_x64_assembler.c` | 每个 `xm_x64.h` 顶层 emit fn 至少 3 个 case，覆盖普通 / 扩展 / 特殊寄存器 |
| arm64 | `tests/unit/jit/test_arm64_assembler.c` | 同上 |
| 共享原语 | `tests/unit/jit/test_x64_modrm.c` | `x64_modrm_mem` 各种 base/disp 边界 |
| Disassembler | `tests/unit/jit/test_<arch>_disasm.c` | round-trip：encode → disasm → expected string |

### 6.3 `llvm-mc` differential

CI 流程：

```text
random program generator
    ↓
emit via xray assembler → bytes_xray
    ↓
disasm via xray xm_<arch>_disasm → text_xray
disasm via `llvm-mc -disassemble` → text_llvm
    ↓
diff text_xray vs text_llvm（normalize whitespace / 立即数格式后）
    ↓
不一致 → fail-fast，dump bytes / both texts / 复现命令
```

`llvm-mc` 仅用于 CI，不是 xray runtime 依赖。差异如果是 xray disassembler 的格式问题，修 disassembler；如果是 assembler emit 错误，修 assembler。

### 6.4 Fuzz 策略

| 层 | 输入 | 检查 |
|---|---|---|
| Assembler 层 fuzz | random `(reg, imm, mem-form)` 组合 | encode → disasm round-trip 一致 |
| Codegen 层 fuzz | random small `Xm` IR program | JIT 编译成功 + 真机执行不 crash + 与 interpreter 结果一致 |
| Driver-level invariant fuzz | random IR op sequence | 每条 IR op emit ≥ 1 字节（commit 8b06997 类回归捕获） |

每 PR 跑 1k 种子；nightly 跑 1M 种子。

## 7. Codegen Driver 契约

### 7.1 IR op switch 必须穷尽

```c
/* 反例（commit 8b06997 的根因模式） */
switch (op) {
    case OP_ADD: ...; break;
    case OP_SUB: ...; break;
    case OP_RT_FOO: /* 沉默 fallback，没 emit 任何字节 */ break;
}
/* → 编译期不报错，runtime 缺字节，下游 STATUS_HEAP_CORRUPTION */
```

```c
/* 正例 */
switch (op) {
    case OP_ADD: ...; break;
    case OP_SUB: ...; break;
    case OP_RT_FOO: emit_rt_helper_call(...); break;
    default:
        XR_UNREACHABLE("unhandled XmOp %d in x64 codegen", op);
}
```

每个对 `XmOp` 的 switch 必须有 `default: XR_UNREACHABLE`，**不允许默认 fallthrough**。

### 7.2 字节数自检

driver 对每个 IR op 的 emit 包一层：

```c
static void emit_xm_op_x64(CodegenCtx *ctx, XmIns *ins) {
    uint32_t pos_before = x64_buf_offset(&ctx->buf);
    emit_xm_op_x64_inner(ctx, ins);
    uint32_t pos_after = x64_buf_offset(&ctx->buf);
    XR_DCHECK(pos_after > pos_before,
              "XmOp %d emitted 0 bytes at IR %u", ins->op, ins->id);
}
```

Release 构建可以用更轻量的 `XR_CHECK`（即使 `NDEBUG` 也保留检查），保证生产环境 fail-fast。

### 7.3 commit 8b06997 根除路径

把 §7.1 + §7.2 在 `xm_codegen_x64*.c` 全面落地后，commit 8b06997 同类 bug 在编译期或 build-time test 阶段就被捕获：

| 路径 | 捕获时机 |
|---|---|
| switch 漏 case | 编译警告（启用 `-Wswitch -Wswitch-enum`）+ runtime `XR_UNREACHABLE` |
| RT-opcode handler 写错 | byte-level unit test 跑出 0-byte emit `XR_DCHECK` 触发 |

## 8. Runtime-sensitive Paths

### 8.1 范围

下列路径**不抽象**到 assembler 层，仍由 driver 手写：

- call ABI（System V / Win64 / AAPCS64 各自规则）
- deopt（spill / register reconstruction）
- OSR entry
- safepoint poll + stack map
- coroutine suspend / resume

### 8.2 理由

- 这些路径强耦合 runtime data layout（`XrIsolate` / `XrCoro` / GC root）
- 每条 IR op 不是 1:1 emit 到 machine code，而是涉及 metadata 写出
- 跨 backend 一致性靠 **parity test** 捕获，不靠 schema

### 8.3 一致性保障

| 路径 | 保障手段 |
|---|---|
| call ABI | 每个 call 形式（known / c-helper / self-direct）都有 x64 / arm64 双 backend test |
| deopt | x64 / arm64 同一 deopt point 重建后的 VM register state byte-equal |
| OSR | `test_osr_entry_pressure` 多点 / 多参 / spill-pressure |
| safepoint | stack map metadata verifier，GC stress 30 分钟 |
| suspend / resume | 双 backend Debug + Release，重点 `1115_cancel.xr` / `1109_await_any.xr` / `1127_coro_priority.xr` / `1128_yield.xr` |

## 9. 与现有 JIT / AOT 边界

### 9.1 JIT

```text
Xi IR  →  Xm IR  →  passes (regalloc / fold / peephole / ...)  →  codegen driver  →  assembler  →  bytes
```

本架构只规范最后两层（driver / assembler / disassembler）。Xi → Xm、passes、regalloc 不受影响。

### 9.2 AOT

AOT 走 `Xi IR → C source → 系统 C compiler`，与本架构正交。AOT 不调 JIT codegen。

### 9.3 VM

VM bytecode 与 JIT 完全独立，不受影响。
